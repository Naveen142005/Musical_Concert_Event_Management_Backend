Musical Concert Management System

Database Design Document

Version 1.0

October 2025

1. Introduction

The Musical Concert Management System is a comprehensive database solution designed to manage all aspects of musical concert operations. The system facilitates efficient management of events, venues, bands, ticket sales, bookings, payments, and user interactions. This database design document provides a detailed overview of the system's data structure, entity relationships, and constraints that ensure data integrity and support business operations.

The database is designed to support multiple stakeholders including event organizers, venue managers, band members, customers, and system administrators. It enables seamless coordination between different entities involved in concert management, from initial event planning to post-event feedback collection.

2. Entity Descriptions

2.1 Events

Represents musical concerts and performances. Each event has details about the concert including name, banner image, description, slot timings, ticket availability status, creation date, and payment information.

2.2 Venue

Contains information about concert locations including venue name, physical location, seating capacity, pricing structure, and operational status.

2.3 Bands

Stores information about musical bands performing at events, including band name, genre, capacity information, member count, current status, and pricing details.

2.4 User

Represents system users including event organizers, customers, and administrators. Contains authentication credentials and basic profile information.

2.5 User_Details

Extended user profile information including gender, date of birth, location details, country, and profile image.

2.6 Tickets

Manages ticket inventory for events including ticket types, pricing, available quantities, and associated event information.

2.7 Bookings

Records customer ticket purchases including booking date, number of tickets, total amount, and booking status.

2.8 Bookings_details

Detailed breakdown of bookings showing specific ticket types and quantities purchased.

2.9 Payments

Tracks payment transactions for bookings including payment method, date, and current status.

2.10 payment_details

Stores detailed payment information associated with specific bookings and users.

2.11 payment(audience)

Records payments made by audiences to organizers for event bookings.

2.12 payment(organizer)

Manages payments from organizers to various service providers and vendors.

2.13 refund

Handles refund requests and processing including refund reason, amount, date, and status.

2.14 feedback

Collects user feedback and reviews for events including ratings, comments, and submission dates.

2.15 Queries

Manages user inquiries and support requests with priority levels and status tracking.

2.16 Role

Defines system roles for access control and permission management.

2.17 booked_snacks

Manages snack and refreshment bookings for events including items ordered and pricing.

2.18 snacks

Catalog of available snacks and refreshments with pricing information.

2.19 Decorations

Lists available decoration options for events including types, pricing, and availability status.

2.20 event_status_history

Maintains historical records of event status changes for audit and tracking purposes.

2.21 facilities_selected

Tracks selection of various facilities including venues, bands, decorations, and booked snacks for events.

2.22 recent_activity

Logs recent user activities and actions within the system for monitoring and analytics.

4. Relationships Between Entities

4.1 One-to-Many Relationships

User to Events: One user (organizer) can create and manage multiple events. Each event belongs to one user. This relationship is implemented through the user_id foreign key in the Events table.

User to Bookings: One user can make multiple bookings for different events. Each booking is associated with one user. The user_id in the Bookings table establishes this relationship.

User to Queries: One user can submit multiple queries or support requests. Each query belongs to one user through the user_id foreign key in the Queries table.

User to User_Details: Each user has one detailed profile record. This one-to-one relationship is maintained through user_id in the User_Details table.

Role to User: One role can be assigned to multiple users. Each user has one role. The role_id in the User table implements this relationship.

Events to Tickets: One event can have multiple ticket types with different pricing. Each ticket type belongs to one event through the event_id foreign key.

Events to Bookings: One event can have multiple bookings from different customers. Each booking is for one specific event, established by event_id in Bookings.

Events to Feedback: One event can receive multiple feedback entries from different attendees. Each feedback entry relates to one event through event_id.

Events to event_status_history: One event can have multiple status change records tracking its lifecycle. Each history entry belongs to one event.

Events to facilities_selected: One event can have one set of selected facilities. This relationship links the event to its chosen venue, band, decorations, and snacks.

Bookings to Bookings_details: One booking can contain multiple ticket types with different quantities. Each booking detail record belongs to one booking through booking_id.

Bookings to Payments: One booking is associated with one payment transaction. The booking_id in the Payments table establishes this relationship.

Payments to refund: One payment can have one refund request if cancellation occurs. The payment_id in the refund table links to the original payment.

User to payment_details: One user can have multiple payment detail records for different transactions. Each payment detail belongs to one user.

4.2 Many-to-Many Relationships

Events and Venues (through facilities_selected): Multiple events can be held at multiple venues over time. The facilities_selected table serves as a junction table with venue_id and event_id.

Events and Bands (through facilities_selected): Multiple events can feature multiple bands, and bands can perform at multiple events. The facilities_selected table links events to bands.

Events and Decorations (through facilities_selected): Multiple events can use multiple decoration setups. The facilities_selected table manages this relationship through decor_id and event_id.

Events and Snacks (through facilities_selected and booked_snacks): Multiple events can order multiple snack combinations. The facilities_selected table links to booked_snacks, which contains the snack items.

4.3 Additional Relationships

payment_details to payment(audience): Payment details are linked to audience payments through payment_details_id, tracking payments from customers to organizers.

payment_details to payment(organizer): Payment details are linked to organizer payments through payment_details_id, tracking payments from organizers to service providers.

Events to payment(audience): Events are linked to audience payments, showing which payments were made for specific events.

Events to payment(organizer): Events are linked to organizer payments, tracking payments made by event organizers for facilities and services.

6. Normalization

The Musical Concert Management System database is designed following normalization principles up to Third Normal Form (3NF) to eliminate data redundancy and ensure data integrity.

6.1 First Normal Form (1NF)

All tables in the database satisfy 1NF requirements. Each table has a primary key that uniquely identifies each record. All attributes contain atomic values with no repeating groups. For example, the Events table stores one event per row with atomic values for name, description, slot, and status. The booked_snacks table uses an array data type for snacks, which is handled appropriately by the database management system.

6.2 Second Normal Form (2NF)

The database satisfies 2NF as all non-key attributes are fully functionally dependent on the entire primary key. There are no partial dependencies. For instance, in the Bookings_details table, attributes like ticket_type, quantity, and unit_total depend on the complete primary key (booking_details_id) and not on part of it. Similarly, in the facilities_selected table, all facility references depend on the primary key, not on portions of it.

6.3 Third Normal Form (3NF)

The database achieves 3NF by eliminating transitive dependencies. No non-key attribute depends on another non-key attribute. User information is separated into User and User_Details tables to avoid transitive dependencies. Payment information is properly separated into Payments, payment_details, payment(audience), and payment(organizer) tables, ensuring that payment attributes depend only on their respective primary keys. Event status history is maintained in a separate table rather than storing historical data in the Events table. The facilities_selected table serves as a proper junction table, referencing primary keys from related tables rather than storing redundant facility information.

6.4 Benefits of Normalization

The normalized structure provides several benefits including reduced data redundancy as information is stored once and referenced through foreign keys, improved data integrity through proper constraint enforcement, easier database maintenance with changes required in only one location, efficient query performance with properly indexed tables, and scalability with a structure that can accommodate growth without major redesign.

7. Conclusion

The Musical Concert Management System database design provides a comprehensive and robust foundation for managing all aspects of musical concert operations. The design effectively addresses the complex relationships between events, venues, bands, users, bookings, payments, and feedback through a well-structured relational model.

Key strengths of this database design include comprehensive event management capabilities that track events from creation through completion with full historical records, flexible user management supporting multiple roles including organizers, customers, and administrators with appropriate access controls, detailed booking and ticketing systems that handle multiple ticket types, quantities, and pricing structures, robust payment processing with separate tracking for audience and organizer payments, complete refund handling, and integrated feedback collection mechanisms for quality improvement.

The database design follows industry best practices with normalization up to 3NF, proper use of primary and foreign keys, appropriate data types and constraints, clear separation of concerns across tables, and scalable architecture supporting system growth. The entity relationship structure ensures data integrity through carefully designed foreign key constraints, prevents data anomalies through normalization, supports complex queries through proper table relationships, enables efficient data retrieval with appropriate indexing opportunities, and facilitates future enhancements with flexible design.

This database design successfully supports all core business operations of the Musical Concert Management System including event planning and setup, venue and band management, ticket sales and bookings, payment processing and refunds, user authentication and authorization, feedback and quality management, and reporting and analytics. The implementation of this database design will provide a solid foundation for building a reliable, efficient, and scalable Musical Concert Management System.
